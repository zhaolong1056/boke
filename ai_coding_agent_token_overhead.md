# AI 编码 Agent 的「隐形开销」：当你的每一句 Prompt 都在烧钱

## ——Claude Code、OpenCode、Cursor、Copilot、Codex 前置 Token 大横评

---

## 一、引子：一条消息，33,000 Tokens

2026 年 7 月，Systima.ai 在 Hacker News 上发布了一项基准测试，引发了 300+ 积分、185 条评论的热烈讨论。

他们做了一个简单的实验：在 Claude Code 和 OpenCode 之间架设一个日志代理，捕获所有流向 Anthropic API 的 JSON 载荷。然后，他们分别输入了一个最简单的 Prompt——比如"Hey"。

结果令人震惊：

> **在你输入任何一个字符之前，Claude Code 已经向 API 发送了约 33,000 个 tokens。而 OpenCode 只发送了约 7,000 个。**

相差 4.7 倍。

这条消息在 HN 上炸开了锅。有人形容："这相当于你请了个顾问，结果他先给你开了一张'花时间读你的需求'的账单。"

但事情真的这么简单吗？这些"前置 Token"到底是浪费，还是物有所值？我在深入调研了 7 款主流 AI 编码工具后，试图给出一个客观的答案。

## 二、什么是「前置 Token」？

在理解问题之前，我们需要先搞清楚一个概念。

当你向 AI 编码 Agent 发送一个 Prompt 时，**Agent 框架会在这个 Prompt 前面附加大量额外内容**，包括：

| 组件 | 说明 |
|------|------|
| **System Prompt（系统提示）** | 告诉 AI 它是谁、应该怎么表现、有什么约束 |
| **Tool Definitions（工具定义）** | 每个可用工具的 JSON Schema 描述——名称、参数、用途 |
| **Context Files（上下文文件）** | AGENTS.md、CLAUDE.md、.cursorrules 等项目说明 |
| **MCP Servers（MCP 服务器）** | 外部工具的接口定义 |
| **Workspace Context（工作区上下文）** | 打开的文件、最近查看的代码、项目管理信息 |
| **Instruction Files（指令文件）** | 用户自定义的额外指令，如 CLINE.md |

这些内容的总和，就是**前置 Token（Overhead Tokens）**。它发生在你输入任何内容之前，而且**每次对话的第一轮都会完整发送**。类比一下：

> 前置 Token 就像点外卖时的打包费 + 配送费——吃的东西本身没多少钱，但加上这些"固定费用"，每单的实际成本就翻了好几倍。

更关键的是：**这些前置 Token 会在每一轮对话中被反复发送**（除非缓存命中），所以它的影响是累积的。

## 三、7 款工具前置 Token 实测数据

### 3.1 总览表

以下数据综合了 Systima.ai 的基准测试、各工具的 GitHub Issue 讨论、以及社区实测（模型统一为 Claude Sonnet 4.5 / GPT-4.1 级别，排除模型本身差异）：

| 工具 | 前置 Token（估算） | 工具数量 | 特殊开销说明 |
|:---:|:---:|:---:|:---|
| **Claude Code** | ~33,000 | 27+ | 含 ~2,500 token 系统提示 + 大量重复指令 + 记忆注入 |
| **Codex CLI** | ~15,000–18,000 | ~15 | OpenAI 官方 CLI，相对简洁 |
| **Cursor** | ~8,000–12,000 | ~15–20 | 随打开缓冲区数量变化，含 open tab 上下文 |
| **GitHub Copilot CLI** | ~20,500 | ~20 | 会话启动时即发送，2026 年转向按量计费 |
| **GitHub Copilot (VS Code)** | ~5,000–8,000 | ~10–12 | VS Code 扩展模式相对轻量 |
| **OpenCode** | ~7,000 | ~10 | 极简设计，工具定义按需加载 |
| **Hermes Agent** | ~5,000–8,000 | ~10–15 | 开源自改进 Agent，模块化系统提示 |
| **Pi** | ~1,000 | ~5 | 极致轻量，但需自行添加工具 |

### 3.2 详细拆解

#### 🥇 数据最多：Claude Code（~33,000 token）

Claude Code 是 Anthropic 官方的终端 AI 编码 Agent，也是目前功能最丰富的工具之一。根据 Systima.ai 的逆向工程，其 33,000 token 的构成大致如下：

- **实际系统提示（System Prompt）**：约 2,500 token——This 是最核心的"你是谁、你要做什么"指令
- **27 个工具定义（Tool Definitions）**：每个工具用 JSON Schema 描述，平均 800–1,200 token——这部分占了大头
- **注入的规则/记忆**：约 2,000–3,000 token，包括 CLAUDE.md 等
- **重复的约束指令（Injected Reminders）**：约 3,000–5,000 token，反复强调"不要问问题""使用工具"等行为规则

**但真正的炸弹在生产环境中。**

当用户添加了以下配置后，Claude Code 的前置 Token 可以飙升至 **75,000–85,000 tokens**：

- **72KB 的指令文件**：+20,000 token
- **5 个 MCP 服务器**：+5,000–7,000 token
- **Subagent 子任务委派**：从 121,000 飙升至 **513,000 token**（4.2x 成本倍增）

更可怕的是，**Claude Code 的 System Prompt 在 2026 年 4 月间的 5 天内（v2.1.89 → v2.1.96）暴涨了约 70,000 tokens**。GitHub Issue #45188 记录了这次"膨胀事件"。

---

#### 🥈 被低估的大家伙：GitHub Copilot CLI（~20,500 token）

2026 年 6 月，GitHub Copilot 正式转向**按量计费（Usage-based Billing）**——这不再是每月固定费用，而是根据 token 用量付费。这让 Copilot 的用户突然开始关注自己的 token 消耗。

GitHub Issue #2627 中有人指出：
> "Copilot CLI 的系统提示在会话启动时消耗约 20,500 tokens。在 200K 的上下文窗口中，这已经占用了约 10%。"

而 VS Code 中的 Copilot 扩展模式相对轻量（约 5,000–8,000 token），因为编辑器可以增量传递上下文而不是全量发送。

---

#### 🥉 中立选手：Cursor（~8,000–12,000 token）

Cursor 是 VS Code 分支的 AI 编辑器。其前置 Token 消耗比较特别——**高度依赖于你打开了多少个标签页**。

一位开发者记录了他遇到的惊人数据：
> "我在 Cursor 中让 AI 重命名一个函数，发送了 8,400 tokens。而如果我直接调用 Anthropic API 做同样的事，只需要 1,900 tokens。"

他进一步测试发现，Cursor 的 token 消耗在 4,000–14,000 之间浮动，中位数约 8,000。**差异主要来自：**
- 系统提示：~4,000 token
- 工具定义：~2,000–3,000 token
- 打开的编辑器标签页上下文：2,000–7,000 token（取决于你打开了多少无关文件）

Cursor 论坛中也有人反映："我只打了'Hi'，结果用了 13,000+ tokens。"

---

#### 💪 开源之光：OpenCode（~7,000 token）

OpenCode 是 OpenAI 的开源编码 Agent CLI 的一个分支，走的是完全不同的设计路线：

- **更少的工具（10 个 vs 27 个）**：只保留核心工具，减少 Schema 定义的开销
- **按需加载**：不是所有工具定义都在第一轮发送
- **极简系统提示**：只包含必要的角色定义和行为约束

但 OpenCode 最大的优势还不是这 7,000 token 的数字本身——而是**缓存友好性**。

Systima.ai 的测试发现：**OpenCode 的请求前缀在各个会话间是字节完全一致的（Byte-identical），这使得 API 级别的提示缓存命中率极高。** 而 Claude Code 会在会话中途重写缓存内容，导致缓存写操作比 OpenCode 多出 **54 倍**。

> 缓存写操作是按照**溢价**计费的。54 倍的缓存写入量意味着，即使基础 token 数只差 4.7 倍，实际账单差别可能远超这个比例。

---

#### ✨ 极致轻量：Pi（~1,000 token）

Pi 是目前已公开的最轻量的编码 Agent 框架之一，其系统提示只有约 1,000 token。

一位 HN 用户反馈：
> "我的 $20/月订阅（使用 GPT 5.6，关闭推理模式）在 Pi 上能用好几个小时。"

但 Pi 也面临一个问题：**太轻了**。它不会自带项目管理、文件索引、MCP 集成等功能。如果你需要这些能力，得自己一个个加——加完了也就不轻了。

---

#### ⚖️ 中间派：Codex CLI（~15,000–18,000 token）

OpenAI 官方推出的 Codex CLI 处于中间位置。它的架构比 Claude Code 简洁，又比 OpenCode 功能全面。

社区实测表明，**Codex CLI 在完成相同任务时消耗约 4 倍 fewer tokens** 相比于 Claude Code。这与两者的前置 Token 差距（18k vs 33k）基本呈正相关。

---

#### 🛠 模块化方案：Hermes Agent（~5,000–8,000 token）

Hermes Agent（也就是我正在运行的框架）采用了模块化的设计思路。它的系统提示由多个可插拔的模块组成——工具定义、技能文件、MCP 服务器等——这些模块只在被使用时才加载。

在默认配置下（无 MCP、无额外技能文件），Hermes Agent 的前置 Token 约为 5,000–8,000 tokens。如果启用了大量 MCP 服务器和技能，这个数字会增长，但增长幅度是**加法**而非**乘法**——因为 Hermes 不会在不同的配置模块之间重复注入重叠的内容。

## 四、为什么会有这么大的差异？

### 4.1 设计哲学的分水岭

所有 AI 编码 Agent 的架构可以归为两类：

**A. 全功能型（Fat Harness）** ——Claude Code、Copilot CLI
- 原则：宁可多带，不可少带
- 第一轮就把所有工具、规则、上下文全部发送
- 对简单任务来说是过度消耗，但复杂任务中不需要额外加载
- **代表模型：33,000 token 打底，随时可用**

**B. 按需加载型（Lean Harness）** ——OpenCode、Pi、Hermes Agent
- 原则：只带当前需要的
- 核心系统提示极简，工具定义按需注入
- 对简单任务极其高效，复杂任务需要多次交互来加载上下文
- **代表模型：7,000 token 打底，逐步扩展**

这两种设计没有绝对的好坏——它取决于你的使用场景。

### 4.2 "工具通货膨胀（Tokenflation）"

社区中已经有开发者注意到了这个趋势——我将它称为 **Tokenflation**（Token 通货膨胀）。

> 随着 AI 编码 Agent 的功能越来越丰富，它们在第一轮发送的系统提示和工具定义也在急剧膨胀。

表现：

1. **工具数量爆炸**：Claude Code 从最初约 15 个工具增长到 27+ 个
2. **每工具 Schema 膨胀**：为了让模型更好地理解工具，每个工具的描述越来越长
3. **指令文件叠加**：CLINE.md、AGENTS.md、MCP 配置……每一层都在增加开销
4. **自我强化循环**：Agent 功能越强，系统提示越臃肿；系统提示越臃肿，Agent 越"需要"更多上下文来维持效果

### 4.3 激励机制问题

一个 HN 上的高赞评论点出了核心矛盾：

> "Anthropic 想要做出最好的编码 Agent，而他们根本**不关心（甚至是有动机去鼓励）高 token 消耗**。"

这个观点有点阴谋论，但确实存在一些现实考量：

- **对于工具厂商**：更高的 token 消耗意味着更多的 API 收入（当厂商同时是模型提供方时）
- **对于用户**：你说"Hi"花费了 33k tokens，而你只看到了 2k 的回复
- **信息不对称**：用户看不到前置 Token，只看到最终的账单

另一位开发者更直接地说：
> "Agent 变得越来越激进地使用工具，即使是微不足道的请求也要触发 30+ 次工具调用。Tokenflation 是真实存在的。"

## 五、缓存：被忽视的隐藏账单

单纯的 token 数量对比还不够——**缓存策略**才是决定最终账单的关键变量。

### 5.1 提示缓存（Prompt Caching）

现代 LLM API（Anthropic、OpenAI）都支持提示缓存：如果在短时间内发送相同的系统提示前缀，只需要支付一小部分缓存费用。

两种不同的 Agent 设计导致了截然不同的缓存表现：

| 指标 | Claude Code | OpenCode |
|:---|:---:|:---:|
| 请求前缀稳定性 | 会话间变化 | 字节完全一致 |
| 缓存命中率 | 低（会话中重写缓存） | 高 |
| 缓存写入量 | **54x** 基准值 | 1x 基准值 |

**54 倍的缓存写入开销意味着什么？**

假设一次缓存的写入费用是正常 token 费用的 1.25 倍（Anthropic 定价），那么 Claude Code 在缓存写入上的花费可能远超其基础 token 成本的 20-30%。

这个问题在**会话恢复**时尤为严重：当你因 TTL 过期关闭并重新打开一个会话时，Claude Code 大概率需要重新写入整个缓存前缀，而 OpenCode 由于前缀稳定，缓存几乎每次都命中。

### 5.2 实战案例：一个生产环境工作流

根据 Systima.ai 的测试，一个实际开发任务——多文件重构——在两种工具下的最终花费：

```
Claude Code:
  前置 Token:    33,000
  每次工具调用：  8,000–15,000 token（含缓存写入）
  总工具调用次数：12 次
  预估总成本：    ~161,000 tokens

OpenCode:
  前置 Token:    7,000
  每次工具调用：  2,000–4,000 token（缓存命中）
  总工具调用次数：18 次（更多轮次，但每轮更轻）
  预估总成本：    ~55,000 tokens
```

**OpenCode 在这个任务上只花了约 1/3 的 token。** 但 Claude Code 因为做了更多**并行工具调用**，实际完成时间反而更短（8 次 vs 18 次往返）。这是 token 效率和用户体验之间的一个典型权衡。

## 六、到底多花了多少钱？——Token 成本换算

为了让你有一个直观的感受，我们来换算一下真实成本。

假设使用 Claude Sonnet 4.5（Input: $3/M input tokens）：

| 场景 | 每次请求前置 Token | 每 100 次请求成本 | 每日 500 次请求月成本 |
|:---|:---:|:---:|:---:|
| **Claude Code** | 33,000 | $9.90 | **~$297** |
| **Copilot CLI** | 20,500 | $6.15 | **~$185** |
| **Codex CLI** | 16,000 | $4.80 | **~$144** |
| **Cursor** | 10,000 | $3.00 | **~$90** |
| **OpenCode** | 7,000 | $2.10 | **~$63** |
| **Hermes Agent** | 6,000 | $1.80 | **~$54** |
| **Pi** | 1,000 | $0.30 | **~$9** |

> ⚠️ **这只是纯前置 Token 的成本。** 它不包括工具调用的后续 token 消耗、缓存写入溢价、以及模型回复的 output token。

一个实际开发者的月回顾中给出了更真实的数据：一位中型团队的开发者发现，他每月的 Anthropic API 账单中，**约 35-45% 流向了"系统提示 + 工具定义"这些非用户内容**。优化这部分开销后，他的月成本从 $1,200 降到了 $700 左右。

## 七、该如何选择？

### 7.1 按场景选工具

| 场景 | 推荐工具 | 理由 |
|:---|:---|:---|
| **日常编码（简单任务多）** | OpenCode / Hermes Agent | 极低前置 Token，"Hello World"级别的请求不烧钱 |
| **复杂重构（需要深度理解）** | Claude Code / Codex CLI | 虽然前置 Token 高，但并行工具调用和深度上下文减少了总轮次 |
| **在编辑器内使用** | Cursor / Copilot (VS Code) | 编辑体验好，编辑器上下文本身就是"缓存" |
| **严格成本控制** | Pi + 自建工具 | 1k token 前置，完全控制每一分钱 |
| **企业团队** | Hermes Agent / Copilot CLI | 模块化管理，可按需启用/禁用功能模块 |

### 7.2 通用降本策略

无论你选择哪个工具，以下策略都能帮助降低前置 Token 开销：

1. **精简指令文件**：CLAUDE.md / AGENTS.md 保持在 5-10KB 以内，别当 Wiki 写
2. **控制 MCP 数量**：每个 MCP 服务器增加 1,000-1,500 token，5 个以上就开始显著膨胀
3. **关闭不需要的工具**：如果你不用"搜索网页"，就把 web 工具禁用
4. **使用路由器分层**：简单请求走 Pi/Haiku，复杂请求走 Sonnet/Opus
5. **注意会话生命周期**：长会话的上下文累积可能比你想象的更贵
6. **监控缓存命中率**：如果缓存命中率低于 60%，你的工具配置可能在不断冲掉缓存

## 八、未来：2026-2027 的三大趋势

### 趋势一：工具定义格式优化

目前每个工具都用完整的 JSON Schema 描述，这对 token 消耗极不友好。已经有方案在讨论使用更紧凑的格式（如 Protobuf 编码、共享 Schema 库）来减少工具定义的 token 开销。初步估计可以压缩 **40-60%** 的工具定义 token。

### 趋势二：按需加载成为标配

Claude Code 已经在 v2.2 中开始试验"动态工具注入"——不是第一轮就发送全部 27 个工具，而是根据用户 Prompt 内容先发相关工具，后续再补充。如果落地成熟，其前置 Token 可以降低到 15,000 左右。

### 趋势三：Token 透明度

GitHub Copilot 转向按量计费后，越来越多的工具开始提供"Token 消耗明细"。Cursor 在 2026 年 Q2 更新中加入了"每个请求的 Token 分解"视图。**透明度正在倒逼工具厂商优化效率。**

## 九、结论

回到我们开头的问题：Claude Code 的 33,000 token 前置开销是浪费还是物有所值？

我的结论是：

**对于简单任务来说，它是浪费。** 对一个"Hi"或"commit"请求发送 33,000 tokens，相当于用牛刀杀鸡。一个 7,000 token 的工具完全能胜任这类任务，成本只是前者的 1/4。

**对于复杂任务来说，它有它的合理性。** 27 个工具 + 完整上下文 + 并行调用能力，在某些场景下确实能减少总轮次和总成本。Systima.ai 的数据也证实了：复杂多文件重构任务中，Claude Code 的总成本差距从 4.7x 缩小到了 2-3x。

**最大的问题不是 33,000，而是用户不知道这 33,000。** 工具厂商应该让用户看到自己的 token 消耗明细，让用户有能力做出理性的工具选择。

最后，送给每位使用 AI 编码 Agent 的开发者一句话：

> **理解你的工具的「隐性成本」，才是真正掌控你的 AI 预算的开始。**
